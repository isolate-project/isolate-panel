## 🏗️ Архитектурное видение

### Принципы

1. **Один сервер = одна панель** — никаких multi-server кластеров
2. **Localhost-only** — панель слушает только `127.0.0.1:8080`, доступ исключительно через SSH-туннель
3. **Минимум ресурсов** — работает на VPS с 1 CPU / 1 GB RAM
4. **Монолит** — все компоненты в одном Docker-контейнере (Supervisord управляет процессами)
5. **Stateless API** — JWT-аутентификация, без серверных сессий
6. **Adapter pattern** — каждое ядро реализует `CoreAdapter` + `StatsClient`, новые ядра добавляются одним файлом

### Взаимодействие панели и ядер

Каждое ядро реализует интерфейс `CoreAdapter`:

| Метод | Назначение |
|---|---|
| `GenerateConfig()` | Генерация нативного конфига (JSON/YAML) из моделей БД |
| `ValidateConfig()` | Валидация конфига перед записью |
| `WriteConfigToDir()` | Запись конфига на диск |
| `CreateStatsClient()` | Создание клиента статистики |
| `SupportedProtocols()` | Список поддерживаемых протоколов |
| `GetHealthCheckEndpoint()` | Endpoint для проверки здоровья |

Управление жизненным циклом ядер:

```
CoreManager ──XML-RPC──▶ Supervisord
   │                        │
   ├── StartCore()          ├── startProcess()
   ├── StopCore()           ├── stopProcess()
   ├── RestartCore()        └── restartProcess()
   └── GetCoreStatus()      └── getProcessInfo()
                              │
                              ▼
   ConfigService ───────▶ Генерация конфига → Запись на диск → Перезапуск ядра
```

Сбор статистики:

| Ядро | API | Порт | Протокол |
|---|---|---|---|
| Xray | StatsService (gRPC) | 10085 | gRPC (dokodemo-door) |
| Sing-box | Clash API | 9090 | HTTP (с API Key) |
| Mihomo | Clash API | 9091 | HTTP (с API Key) |

## 🔧 Текущая фаза разработки

### Фокус: HAProxy / SNI-роутинг

**Приоритет:** Добавление HAProxy для продвинутого роутинга трафика.

Сейчас каждое ядро слушает на уникальных портах. Это работает, но создаёт ограничения:
- Каждый инбаунд требует уникальный порт
- Невозможно мультиплексировать несколько протоколов на одном порту
- Нет SNI-based и path-based роутинга

**HAProxy решит:**
- 🔀 Несколько инбаундов на одном порту (SNI-based routing)
- 🛤️ Path-based routing для HTTP/WebSocket
- ⚖️ Централизованный rate limiting
- 📊 Единая точка мониторинга соединений

## 📌 Замечания по архитектуре

### Сильные стороны

1. **Adapter pattern для ядер** — добавление нового ядра требует только 1 адаптер + RegisterCore(), без изменений в остальном коде
2. **Protocol registry** — протоколы декларативно описаны, Smart Forms генерируются автоматически
3. **Single-container deployment** — упрощает установку и поддержку
4. **SSH-tunnel-only** — атакующая поверхность минимальна
5. **Lazy core start** — экономия ресурсов на слабых VPS

### Потенциальные риски

1. **SQLite при высокой нагрузке** — WAL mode помогает, но при 100+ одновременных соединениях возможны задержки записи. Mitigation: batch-записи в TrafficCollector
2. **Supervisord как PID 1** — если Supervisord упадёт, весь контейнер «умрёт». Mitigation: `autorestart=true`, Docker restart policy
3. **In-memory кеш подписок** — при рестарте панели кеш сбрасывается, первый запрос будет медленнее. Mitigation: приемлемо для MVP, Ristretto быстро прогревается
